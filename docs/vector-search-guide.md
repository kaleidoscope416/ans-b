# 校园生活百事通向量检索设计说明

## 1. 文档目的

本文档说明校园生活百事通如何实现向量检索，以及为了让向量检索效果更好，知识库和用户投稿需要提供哪些数据。本文档重点服务于用户贡献问答设计、管理员审核设计和后端检索模块实现。

## 2. 向量检索解决什么问题

关键词检索依赖字面匹配，用户表达稍微变化就可能搜不到。

示例：

```text
知识库问题：一食堂营业时间是什么？
用户提问：食堂几点关门？
```

这两个问题没有完全相同的关键词，但语义接近。向量检索会把文本转换成数字向量，再通过相似度计算判断两段文本是否表达相近意思。

因此向量检索主要解决：

1. 同义表达问题，例如“几点关门”和“营业时间”。
2. 口语化提问问题，例如“在哪打印”和“打印服务地点”。
3. 问法不标准问题，例如“校园卡丢了咋办”和“校园卡挂失流程”。
4. 长文档检索问题，例如从学生手册中找到相关片段。

## 3. 基本流程

```text
知识入库
  ↓
整理可检索文本
  ↓
生成文本向量
  ↓
保存到 knowledge_chunks
  ↓
用户提问
  ↓
生成问题向量
  ↓
计算问题向量和知识向量相似度
  ↓
返回最相似的知识片段
```

第一版建议采用混合检索：

```text
最终分数 = 关键词分数 * 0.4 + 向量相似度 * 0.5 + 意图匹配分数 * 0.1
```

这样可以兼顾精确匹配和语义匹配。

## 4. 知识入库需要哪些数据

### 4.1 最小必需数据

一条可检索知识至少需要：

| 字段 | 是否必需 | 作用 |
|---|---|---|
| 标准问题 | 必需 | 用于匹配用户问题 |
| 标准答案 | 必需 | 用于返回最终答案 |
| 状态 | 必需 | 只有通过审核的知识可以被检索 |

示例：

```text
标准问题：一食堂营业时间是什么？
标准答案：一食堂早餐 6:30-9:00，午餐 10:30-13:30，晚餐 16:30-20:00。
状态：approved
```

### 4.2 推荐补充数据

为了提升检索效果，建议补充：

| 字段 | 是否必需 | 作用 |
|---|---|---|
| 分类 | 推荐 | 帮助判断问题领域 |
| 标签 | 推荐 | 增强关键词匹配 |
| 同义问法 | 推荐 | 覆盖更多用户表达 |
| 信息来源 | 推荐 | 方便管理员审核和溯源 |
| 适用范围 | 推荐 | 区分校区、地点、部门 |
| 生效时间 | 可选 | 处理临时通知和过期信息 |
| 备注 | 可选 | 保存审核说明或补充说明 |

示例：

```text
标准问题：一食堂营业时间是什么？
同义问法：
- 食堂几点开门？
- 食堂几点关门？
- 一食堂晚上营业到几点？
分类：餐饮服务
标签：食堂、营业时间、一食堂、开放时间
适用范围：本部校区
信息来源：一食堂门口公告
生效时间：长期有效
```

## 5. 向量文本怎么组织

向量模型接收的是一段文本。不要只把答案丢进去，也不要只把标题丢进去。推荐把问题、同义问法、分类、标签和答案摘要组合成一段“可检索文本”。

推荐格式：

```text
问题：一食堂营业时间是什么？
同义问法：食堂几点开门；食堂几点关门；一食堂晚上营业到几点
分类：餐饮服务
标签：食堂，营业时间，一食堂，开放时间
答案：一食堂早餐 6:30-9:00，午餐 10:30-13:30，晚餐 16:30-20:00。
适用范围：本部校区
```

这段文本用于生成向量，保存到 `knowledge_chunks.chunk_text`，对应向量保存到 `knowledge_chunks.embedding`。

## 6. FAQ 类型知识怎么做向量

FAQ 一般是一问一答，第一版可以做到“一条知识生成一个片段”。

```text
knowledge_items
  - title: 一食堂营业时间
  - question: 一食堂营业时间是什么？
  - answer: 一食堂早餐 6:30-9:00，午餐 10:30-13:30，晚餐 16:30-20:00。
  - category: 餐饮服务
  - tags: 食堂、营业时间、一食堂

knowledge_chunks
  - item_id: 对应知识 ID
  - chunk_text: 问题 + 标签 + 答案组合后的文本
  - embedding: chunk_text 的向量
```

用户问“食堂几点关门”时，系统会生成该问题的向量，再和 `knowledge_chunks.embedding` 做相似度比较。

## 7. 文档类型知识怎么做向量

如果后续上传学生手册、图书馆规则、校历文件，文档通常比较长，不能整篇直接生成一个向量。需要先切片。

建议切片规则：

1. 每个片段 300 到 800 个中文字符。
2. 按标题、段落、条款切分，尽量不要从句子中间切断。
3. 每个片段保留来源标题和章节名。
4. 相邻片段可以保留少量重叠内容，避免上下文断裂。

示例：

```text
文档：图书馆读者须知
章节：借阅规则
片段：本科生一次最多可借 10 本图书，借期 30 天，可续借 1 次...
```

生成向量时推荐组合：

```text
文档：图书馆读者须知
章节：借阅规则
分类：图书馆
内容：本科生一次最多可借 10 本图书，借期 30 天，可续借 1 次...
```

## 8. 用户投稿问答应该收哪些数据

用户投稿的目标不是让用户填写“标准知识库字段”，而是让用户提供足够的信息，方便管理员审核和系统后续检索。

### 8.1 第一版投稿表单字段

建议第一版用户投稿页面收这些字段：

| 字段 | 是否必填 | 用户填写示例 | 作用 |
|---|---|---|---|
| 问题 | 必填 | 体育馆周末开放吗？ | 作为标准问题候选 |
| 参考答案 | 必填 | 周六周日 9:00-21:00 开放，需要刷校园卡进入。 | 作为答案候选 |
| 分类 | 可选 | 场馆服务 | 帮助归类和筛选 |
| 标签 | 可选 | 体育馆、开放时间、周末 | 增强关键词检索 |
| 信息来源 | 可选 | 体育馆门口公告 | 帮助管理员核实 |
| 备注 | 可选 | 我是周五晚上看到的公告 | 保存补充说明 |

这些字段已经能支撑第一版投稿和向量检索。

### 8.2 不建议让用户填写的字段

不要让普通用户填写这些内容：

1. 向量。
2. 检索分数。
3. 知识 ID。
4. 审核状态。
5. 是否进入向量库。
6. chunk 内容。

这些应由系统和管理员处理。

## 9. 为了检索效果，投稿问题应该怎么引导

投稿页面可以用占位提示引导用户写清楚问题。

问题输入框提示：

```text
请写成一个具体问题，例如：体育馆周末开放吗？
```

答案输入框提示：

```text
请尽量写清楚时间、地点、流程、联系人或限制条件。
```

信息来源输入框提示：

```text
例如：学校官网通知、现场公告、辅导员通知、部门老师告知。
```

备注输入框提示：

```text
可以补充你看到信息的时间、地点，方便管理员核实。
```

## 10. 管理员审核时需要做什么

用户投稿不能直接生成向量进入知识库。推荐流程：

```text
学生投稿
  ↓
进入 pending 状态
  ↓
管理员审核
  ↓
管理员修正问题、答案、分类、标签
  ↓
审核通过
  ↓
写入 knowledge_items
  ↓
生成 chunk_text
  ↓
生成 embedding
  ↓
进入正式检索范围
```

管理员审核时重点检查：

1. 问题是否清楚。
2. 答案是否准确。
3. 信息是否过期。
4. 分类是否正确。
5. 标签是否能覆盖常见问法。
6. 是否需要补充同义问法。
7. 是否适合公开给所有学生。

## 11. 同义问法怎么设计

同义问法对向量检索和关键词检索都有帮助。

示例：

```text
标准问题：校园卡丢了怎么办？

同义问法：
- 饭卡丢了怎么办？
- 校园卡怎么挂失？
- 一卡通找不到了怎么办？
- 校园卡补办在哪里？
```

第一版数据库没有单独的同义问法字段时，可以先由管理员把同义问法写入 `chunk_text`，或者放在备注中由系统生成检索文本时合并进去。后续可以扩展 `aliases` 字段。

## 12. 分类和标签建议

为了方便用户投稿和管理员审核，建议第一版固定一组分类。

推荐分类：

1. 校历安排。
2. 餐饮服务。
3. 图书馆。
4. 校园卡。
5. 报修服务。
6. 交通校车。
7. 打印复印。
8. 宿舍生活。
9. 社团活动。
10. 场馆服务。
11. 教务事务。
12. 其他。

标签由用户自由填写或从常用标签中选择。

示例：

```text
分类：打印复印
标签：打印、自助打印、图书馆、教学楼
```

## 13. 检索时如何使用这些数据

用户提问后，系统可以同时做三件事：

### 13.1 意图识别

根据关键词判断问题类型。

```text
包含“几点、开放、关门、营业”：查询时间
包含“在哪、哪里、地点”：查询地点
包含“怎么、流程、如何”：查询流程
包含“电话、联系、负责人”：查询联系方式
```

### 13.2 关键词检索

在问题、答案、分类、标签中查找关键词。

例如：

```text
用户问题：在哪打印？
关键词：打印
匹配字段：分类=打印复印，标签=打印、自助打印
```

### 13.3 向量检索

把用户问题转换为向量，和知识片段向量计算相似度。

例如：

```text
用户问题：饭卡丢了咋办？
命中知识：校园卡丢了怎么办？
```

## 14. 投稿字段和检索字段的关系

| 投稿字段 | 是否进入正式知识库 | 是否参与向量检索 | 说明 |
|---|---|---|---|
| 问题 | 是 | 是 | 核心检索内容 |
| 参考答案 | 是 | 是 | 用于回答和辅助匹配 |
| 分类 | 是 | 是 | 帮助限定领域 |
| 标签 | 是 | 是 | 增强关键词和语义检索 |
| 信息来源 | 可选 | 通常否 | 主要用于审核和溯源 |
| 备注 | 可选 | 通常否 | 主要用于审核补充 |
| 投稿人 | 否 | 否 | 只用于个人中心和审核记录 |
| 审核状态 | 否 | 否 | 控制是否进入检索范围 |

说明：信息来源和备注通常不直接参与回答，避免把未经整理的描述混入答案。但管理员可以把其中有价值的信息整理进正式答案。

## 15. 示例：用户投稿到可检索知识

### 15.1 用户投稿

```text
问题：学校哪里可以打印？
参考答案：图书馆一楼大厅有自助打印机，教学楼 A 区一楼也可以打印。
分类：打印复印
标签：打印、自助打印、图书馆、教学楼
信息来源：同学实际使用
备注：图书馆打印机支持校园卡支付。
```

### 15.2 管理员审核后整理

```text
标准问题：学校哪里可以自助打印？
标准答案：图书馆一楼大厅和教学楼 A 区一楼提供自助打印服务，通常支持校园卡支付。
分类：打印复印
标签：打印、自助打印、图书馆、教学楼、校园卡支付
同义问法：
- 在哪里打印？
- 学校哪里能打印？
- 图书馆可以打印吗？
```

### 15.3 系统生成向量文本

```text
问题：学校哪里可以自助打印？
同义问法：在哪里打印；学校哪里能打印；图书馆可以打印吗？
分类：打印复印
标签：打印，自助打印，图书馆，教学楼，校园卡支付
答案：图书馆一楼大厅和教学楼 A 区一楼提供自助打印服务，通常支持校园卡支付。
```

当用户问“在哪打印”时，系统就更容易命中这条知识。

## 16. 第一版实现建议

第一版不要把向量检索做复杂。建议按以下顺序实现：

1. 先完成用户投稿字段：问题、答案、分类、标签、信息来源、备注。
2. 投稿审核通过后写入正式知识库。
3. 将问题、答案、分类、标签组合成 `chunk_text`。
4. 调用 embedding 模型生成向量。
5. 把向量保存到 `knowledge_chunks.embedding`。
6. 用户提问时生成问题向量。
7. 使用 pgvector 查询最相似的知识片段。
8. 结合关键词分数和意图分数排序。

第一版最重要的是数据结构清楚、审核流程稳定、检索结果可解释。
