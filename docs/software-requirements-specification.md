# 校园生活百事通软件需求规约

## 1. 引言

### 1.1 编写目的

本文档用于说明校园生活百事通系统的软件需求，明确系统目标、用户角色、功能需求、非功能需求和验收要求，为后续设计、开发和测试提供依据。

### 1.2 项目背景

新生入学和学生日常学习生活中，经常需要查询校历、食堂开放时间、图书馆规则、报修流程、社团联系方式、校车时刻表等信息。这些信息通常分散在官网、公众号、学生手册和通知文件中，查询成本较高。

本系统通过建设统一的校园生活知识库，为学生提供自然语言问答服务，并通过用户贡献和管理员审核机制持续更新知识内容。

### 1.3 适用范围

本文档适用于校园生活百事通第一版系统。第一版包括学生桌面客户端、`admin-console` 管理端、云端服务和数据库，不包含移动端 App、多学校管理和复杂官网爬虫。

## 2. 总体描述

### 2.1 产品定位

校园生活百事通是一个面向高校学生的智能问答知识库系统。系统提供统一入口，使学生可以用自然语言查询校园生活信息。

### 2.2 用户角色

| 角色 | 描述 |
|---|---|
| 学生用户 | 注册登录，查询校园生活信息，提交新的问答内容，查看个人投稿历史 |
| 管理员 | 通过 `admin-console` 维护知识库，导入数据，审核用户投稿 |

### 2.3 产品功能概述

系统包含以下功能：

1. 智能检索问答。
2. 热点问题推荐。
3. 学生注册和登录。
4. 个人中心。
5. 用户贡献。
6. 管理员登录。
7. 知识库管理。
8. 批量导入。
9. 投稿审核。
10. 可选的大模型增强回答。

## 3. 功能需求

### 3.1 智能检索问答

学生用户应能够输入自然语言问题，系统应返回最匹配的答案。

功能要求：

1. 系统应支持中文自然语言问题输入。
2. 系统应对问题进行清洗、归一化和意图识别。
3. 系统应支持关键词检索和语义检索。
4. 系统应对候选答案进行排序。
5. 系统应返回答案、匹配问题、置信度和相关问题。
6. 系统应在未命中时给出明确提示。
7. 系统不得在无知识依据时编造答案。

### 3.2 热点问题推荐

系统应根据用户查询行为生成热点问题排行榜。

功能要求：

1. 系统应记录用户查询行为。
2. 系统应按查询次数统计高频问题。
3. 系统应展示热点问题列表。
4. 学生用户应可以点击热点问题查看答案。

### 3.3 学生注册和登录

学生用户应能够注册账户并登录系统。

功能要求：

1. 学生应可以使用账号和密码注册。
2. 注册账号应保持唯一。
3. 学生应可以使用账号和密码登录。
4. 登录成功后学生客户端应保存登录状态。
5. 登录失败时系统应给出明确错误提示。
6. 学生密码不得明文存储。

### 3.4 个人中心

已登录学生应能够在个人中心查看自己的账户信息和投稿历史。

功能要求：

1. 个人中心应展示当前登录学生的基础信息。
2. 个人中心应展示该学生提交过的投稿记录。
3. 投稿历史应展示问题、状态、提交时间和审核备注。
4. 未登录学生访问个人中心时，系统应要求先登录。

### 3.5 用户贡献

已登录学生用户应能够提交新的问答内容。

功能要求：

1. 用户应可以填写问题和参考答案。
2. 用户可选择填写分类、标签、信息来源和备注。
3. 用户提交内容默认进入待审核状态。
4. 未审核内容不得直接公开。
5. 系统应提示投稿提交结果。
6. 投稿记录应绑定当前登录学生。

### 3.6 管理员登录

管理员应通过 `admin-console` 使用账号和密码登录系统。

功能要求：

1. 系统应校验管理员账号和密码。
2. 登录成功后管理员可访问 `admin-console` 管理功能。
3. 登录失败时系统应给出错误提示。
4. 登录状态过期后应要求重新登录。
5. 管理员密码不得明文存储。

### 3.7 知识库管理

管理员应能够维护知识库内容。

功能要求：

1. 管理员应可以新增知识。
2. 管理员应可以编辑知识。
3. 管理员应可以删除或禁用知识。
4. 管理员应可以按关键词、分类、标签和状态筛选知识。
5. 只有审核通过或启用状态的知识可以被学生检索。
6. 知识变更后系统应更新检索内容。

### 3.8 批量导入

管理员应能够批量导入常见问答。

功能要求：

1. 系统应支持通过文件导入 FAQ。
2. 系统应校验导入文件格式。
3. 系统应校验导入数据的必填内容。
4. 系统应返回导入成功数量和失败原因。
5. 合法数据应进入知识库。

### 3.9 投稿审核

管理员应能够审核学生提交的新问答。

功能要求：

1. 管理员应可以查看待审核投稿列表。
2. 管理员应可以查看投稿详情。
3. 管理员应可以审核通过投稿。
4. 管理员应可以驳回投稿。
5. 审核通过后投稿内容应进入知识库。
6. 审核驳回后投稿不得进入知识库。

### 3.10 大模型增强回答

系统可选接入大模型，将检索结果整理为更自然的回答。

功能要求：

1. 大模型增强不应作为基础问答的强依赖。
2. 大模型回答必须基于检索到的知识内容。
3. 当检索结果为空时，系统不应让大模型自由回答。
4. 大模型调用失败时，系统应返回知识库原始答案。

## 4. 非功能需求

### 4.1 性能需求

1. 不调用大模型时，普通问答应尽量在 2 秒内返回。
2. `admin-console` 列表查询应支持分页。
3. 热点问题查询应限制返回数量。

### 4.2 可用性需求

1. 学生客户端只提供学生功能，`admin-console` 只提供管理员功能。
2. 表单提交失败时应显示明确错误原因。
3. 问答未命中时应给出可理解提示。
4. `admin-console` 列表应支持搜索、分页和筛选。

### 4.3 安全需求

1. 管理功能必须进行身份校验。
2. 管理员密码必须加密存储。
3. 学生密码必须加密存储。
4. 用户提交内容需要审核后才能公开。
5. 查看个人中心和提交投稿必须校验学生登录状态。
6. 文件导入需要限制文件类型和大小。
7. 系统不应向普通用户返回数据库错误细节。

### 4.4 可维护性需求

1. 后端应按模块组织代码。
2. 检索逻辑应独立封装。
3. 配置项应集中管理。
4. 日志应记录关键错误和管理操作。

### 4.5 可扩展性需求

1. 系统应支持后续扩展文档导入。
2. 系统应支持后续切换 embedding 模型。
3. 系统应支持后续接入不同大模型。
4. 系统应支持后续扩展 Web 端或移动端。

## 5. 验收需求

系统验收时应至少完成以下场景：

1. 管理员登录 `admin-console`。
2. 管理员在 `admin-console` 新增食堂营业时间知识。
3. 学生提问“食堂几点关门”并获得正确答案。
4. 管理员在 `admin-console` 批量导入多条 FAQ。
5. 学生注册并登录系统。
6. 学生提交一条新问答。
7. 学生在个人中心看到该投稿历史。
8. 管理员在 `admin-console` 审核通过该问答。
9. 审核通过后的问答可以被检索到。
10. 系统展示热点问题排行榜。
11. 系统对未命中问题给出合理提示。
